# IEC 61131-3 (ST) — what ironplc actually accepts

IA2 compiles via the vendored `ironplc` compiler. It targets IEC 61131-3 **Edition 2 Structured Text** by default. This is a working cheat-sheet for *what compiles here*, plus the quirks that bite.

## POU kinds

```iecst
PROGRAM main          ... END_PROGRAM        (* one top-level; the thing you run *)
FUNCTION_BLOCK pid    ... END_FUNCTION_BLOCK  (* stateful, instances retain state *)
FUNCTION  clamp : REAL ... END_FUNCTION       (* stateless, returns a value *)
```

- A `.st` file may declare several POUs side by side.
- **Only one PROGRAM is emitted per compilation** (ironplc `find_program` picks the first). FBs and FUNCTIONs declared alongside are fine and callable; extra PROGRAMs are silently dropped. Don't put two PROGRAMs in a project you intend to `cs run` as a schedule.

## Variable blocks

```iecst
VAR            ... END_VAR     (* internal, retained between scans, NOT across restart *)
VAR_INPUT      ... END_VAR     (* FB/FUNCTION inputs *)
VAR_OUTPUT     ... END_VAR     (* FB/FUNCTION outputs *)
VAR_IN_OUT     ... END_VAR     (* by-reference *)
VAR RETAIN     ... END_VAR     (* persists across power cycle / redeploy — IA2 saves to retain.json *)
VAR CONSTANT   ... END_VAR
VAR_GLOBAL     ... END_VAR     (* at CONFIGURATION/RESOURCE scope *)
```

**`VAR RETAIN` is real in IA2.** Names in a `VAR RETAIN` block are extracted at compile time and their values are snapshotted to `<install_dir or project>/state/retain.json` every 5 s and on clean stop, then restored on next start. Use it for lifetime counters, last setpoints, batch totals. (Caveat: values are restored as i32 — types wider than 32 bits, LREAL/LINT/LWORD, truncate. Fine for counters/setpoints.)

## Types

| Class | Types |
|---|---|
| Bit | `BOOL` |
| Signed int | `SINT` `INT` `DINT` `LINT` |
| Unsigned int | `USINT` `UINT` `UDINT` `ULINT` |
| Real | `REAL` `LREAL` |
| Bit string | `BYTE` `WORD` `DWORD` `LWORD` |
| Time/date | `TIME` `LTIME` `DATE` `TOD` `DT` |
| String | `STRING` (`WSTRING` is upstream-WIP; don't rely on it yet) |

Initialisers: `VAR x : INT := 42; r : REAL := 3.14; t : TIME := T#500ms; flag : BOOL := TRUE; END_VAR`

Multi-name decls work: `VAR a, b, c : BOOL; END_VAR`.

## Statements

```iecst
x := y + 1;                          (* assignment *)

IF cond THEN ... ELSIF other THEN ... ELSE ... END_IF;

CASE selector OF
  1:      action1();
  2, 3:   action2();
  ELSE    fallback();
END_CASE;

FOR i := 0 TO 9 BY 1 DO ... END_FOR;
WHILE cond DO ... END_WHILE;
REPEAT ... UNTIL cond END_REPEAT;

EXIT;        (* break out of a loop *)
RETURN;      (* early return; in a value-returning FUNCTION, set the return var first *)
```

Operators: `+ - * / MOD` · `= <> < <= > >=` · `AND OR XOR NOT` · `**` (power). Booleans use `AND`/`OR`/`NOT`, not `&&`/`||`.

## Standard function blocks (the library ironplc ships)

Timers and edges are the bread and butter of PLC logic:

```iecst
VAR
  t_fill : TON;        (* on-delay timer *)
  rise   : R_TRIG;     (* rising-edge detector *)
  latch  : RS;         (* reset-dominant SR latch *)
  up     : CTU;        (* up counter *)
END_VAR

t_fill(IN := valve_cmd, PT := T#3s);    (* call with named params *)
fill_done := t_fill.Q;                   (* read the output *)

rise(CLK := start_btn);
IF rise.Q THEN starts := starts + 1; END_IF;
```

Common library FBs: `TON` `TOF` `TP` (timers) · `R_TRIG` `F_TRIG` (edges) · `CTU` `CTD` `CTUD` (counters) · `SR` `RS` (latches). Call an FB instance by name with named arguments; read its outputs with `.Q`, `.ET`, etc.

## Calling your own FB

```iecst
FUNCTION_BLOCK safety
  VAR_INPUT guard, curtain : BOOL; END_VAR
  VAR_OUTPUT permit : BOOL; END_VAR
  permit := guard AND curtain;
END_FUNCTION_BLOCK

PROGRAM main
  VAR sfb : safety; ok : BOOL; END_VAR
  sfb(guard := door_closed, curtain := beam_clear);
  ok := sfb.permit;
END_PROGRAM
```

## Quirks that bite (verify with `cs check` if unsure)

- **No two PROGRAMs** in a runnable project (see above).
- **`AND`/`OR`/`NOT`**, never C-style `&&`/`||`/`!`.
- **Statements end with `;`** including the last one before `END_IF`/`END_WHILE`.
- **Time literals** are `T#500ms`, `T#1s`, `T#2m30s` — not bare numbers.
- **Don't write your own `CONFIGURATION`/`RESOURCE`/`TASK` block** in a POU file. IA2 synthesises the CONFIGURATION from `tasks.toml` at compile time and strips any you write in by hand. Schedule via `cs tasks set`, not via ST.
- **`AT %IX0.0`-style direct addressing is ignored.** IA2 binds variables to hardware through `iomap.toml` (the `application`/`variable`/`device`/`channel` table), not IEC located variables. Declare plain variables; map them in the iomap.
- **Empty VAR blocks are allowed** (IA2 sets `allow_empty_var_blocks`) — a freshly-created POU skeleton compiles even before you add variables.

When a compile fails, `cs check --explain FILE` or `cs explain P####` gives the full ironplc problem doc — paste that, don't guess.

## The process-control FB library (`library/process-control/`)

Before writing a PID / alarm / device block from scratch, check the repo's ST library — it covers the standard block set every mainstream DCS/PLC ships, validated against this dialect (`cs check` clean, project-compile clean):

- **Loops**: `FB_PID` (velocity form; ff feedforward, output track, sp_rate ramp, dev output, bumpless auto/manual, anti-windup), `FB_PID_POS` (position form; observable p/i/d parts, SP weighting b, D-on-PV + PT1 filter, back-calculation anti-windup), `FB_PID_3STEP` (3-step pulse PID for motorized actuators; travel-time position estimate when no feedback, min-pulse + deadband), `FB_MANSTATION` (manual/bias station with balanced bumpless transfer + rate limit), `FB_GAIN_SCHED3` (3-zone kp/ti/td scheduling, hysteresis + optional boundary blend — wires into PID param inputs), `FB_RAMPSOAK8` (8-segment ramp-soak setpoint programmer; run/hold/advance, cyclic), `FB_RATIO`, `FB_SELECT_HL` (override), `FB_SPLIT_RANGE`, `FB_PWM` (time-proportioning)
- **Signal**: `FB_SCALE` (NE43 fault), `FB_LAG` (PT1), `FB_LEADLAG`, `FB_RAMP`, `FB_RATE_LIMIT`, `FB_SQRT_FLOW` (DP flow + cutoff), `FB_FLOW_COMP` (gas pressure/temperature compensation — sqrt-DP or linear-volumetric, ideal gas), `FB_PV_HOLD` (bad-PV last-good hold / substitute / pass-through with timeout), `FB_CHAR` (8-point characterizer — scalar pairs, NOT arrays: FB-scope arrays pass `cs check` but fail project codegen)
- **Alarms**: `FB_ALARM_HL` (HH/H/L/LL + deadband + delay + ack), `FB_ALARM_DEV`, `FB_ALARM_ROC`
- **Devices**: `FB_MOTOR`, `FB_MOTOR_REV` (reversing motor; reversal lockout delay, feedback-timeout fault latch), `FB_VALVE`, `FB_MOV` (motor-operated valve open/stop/close; dual limit switches + stroke timeout), `FB_DOSE` (coarse/fine dosing with in-flight cutoff, qty integration + overshoot), `FB_DUTY2` (duty/standby pump alternation), `FB_RUNTIME` (hours+starts), `FB_INTERLOCK8` (first-out capture), `FB_DEBOUNCE`, `FB_HYST`, `FB_HILO_FILL`, `FB_PULSER`, `FB_ALT2`, `FB_TOTALIZER` (RETAIN-ready)
- **Math/dynamics**: `FB_LIMIT` (clamp + q_hi/q_lo, lo>hi misconfig signal), `FB_STATS` (Welford min/max/avg/sd, sample-based), `FB_AVG_TIME` (tumbling-window time average), `FB_INTEG` (general integrator: reset>preset>hold, clamp + flags), `FB_DERIV` (filtered du/dt, auto 3*dt_s), `FB_DEADTIME8` (transport delay — 8 scalar taps + phase interpolation, resolution t_delay/8), `FB_MID3` (median / 2oo3 voting with degraded modes), `FB_CASCADE` (master/slave SP coupling + track back-calculation for bumpless CAS switching)

Usage — import via the library API (don't hand-copy into `pous/`):

```bash
GET    /api/library                  # registry + per-project imported state
POST   /api/library/import          # {"library":"process-control","blocks":["fb_pid.st"]} — empty blocks = all
DELETE /api/library/<name>          # remove from project
# or: scripts/import-fb-library.sh <project-name-or-dir> [fb_pid fb_alarm_hl ...]
```

Imported blocks land in `pous/lib/process-control/` (vendored copies — the project stays self-contained for edge deploys) and the import is recorded in project.toml `[libraries]`. The `lib/` subtree is read-only through the generic POU routes (409) — to modify a block, copy it out of `lib/` and own the copy. Re-importing overwrites = update. The IDE shows the subtree as a separate "Libraries" section; `/api/check` resolves cross-file FBs against the whole open project, so editor diagnostics on POUs that call library blocks are real (per-file `cs check` still can't cross-resolve — trust the editor/project compile).

Every block uses the `dt_s : REAL` manual-time convention — pass the task interval in seconds. `library/process-control/README.md` has the per-block index, vendor equivalents, and the full dialect-gotcha list (read it before writing new FBs).
